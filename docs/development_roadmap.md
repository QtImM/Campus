# HKCampus: AI 驱动开发路线图

本项目分为两个阶段，旨在构建一个生产级的校园应用，同时在 AI 工程化面试中展现深度。

---

## 🏗️ 第一阶段：业务闭环与主动智能 (MVP)
**技术栈**: React Native (Expo) + TypeScript + Supabase + 大模型 (DeepSeek/OpenAI)
**当前重点**: 快速交付、提升用户体验、打通核心业务流。

### 1. 核心功能落地
- [x] **高精度导航**: 利用 GPS 坐标实现精确的校园地图标记。
- [x] **换课功能 (真实数据)**: 实现多选 Section、Supabase 实时数据同步。
- [ ] **即时通讯/评论**: 基于 Postgres Changes 的低延迟社交互动。

### 2. 主动式 AI 集成 (Agent 叙事)
- **功能**: **“换课代理人 (Matcher Agent)”**。
- **技术**: Supabase Edge Functions (TS) + LLM。
- **流程**: 
    1. 当有新的换课请求插入数据库时，触发 Webhook。
    2. Edge Function 调用大模型，根据用户的“自然语言偏好”进行语义匹配。
    3. 系统主动向符合条件的用户推送“智能消息”，而非被动等待搜索。
- **面试价值**: 展示“事件驱动型 AI”落地能力，证明你不仅会调 API，还能将其嵌入核心业务流。

---

## 🚀 第二阶段：技术深度与微服务架构
**技术栈**: TS/Supabase + **Python (FastAPI)** + 向量数据库 (Vector DB) + OCR 识别
**目标**: 解决高复杂度的 AI 问题，展现架构设计水平。

### 1. “AI 动力引擎” (Python 微服务)
- **角色**: 处理 Python 具有生态优势的重型任务（数学、AI 算法、计算机视觉）。
- **通信**: Supabase 作为共享状态中心，前端或 Edge 函数异步调用 Python API。

### 2. 关键技术突破点
#### A. 基于图论的最优匹配算法
- **场景**: 当 100 个人同时想换课时，存在多方交换链（A->B->C->A）。
- **逻辑**: 使用 Python (NetworkX) 编写图算法，寻找最优交换闭环，解决 NP-Ref 级别的撮合难题。

#### B. 智能课表解析 (OCR)
- **场景**: 手动输入课程太慢。
- **逻辑**: 使用 **PaddleOCR** 或 **EasyOCR** 解析校园官网的课表截图，自动填充用户的“拥有课程”列表。

#### C. 语义搜索与 RAG
- **逻辑**: 在 Supabase 中使用 `pgvector` 存储课程评论的向量。Python 负责计算 Embedding，实现“推荐给分好且没有考试的课”这类模糊搜索。
